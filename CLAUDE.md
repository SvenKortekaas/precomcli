# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A zero-dependency Node.js CLI client for the PreCom Mobile API (`https://pre-com.nl/Mobile`), a pager/alarm/scheduling system. Entry point: `bin/precomcli.js` → `src/cli.js`.

## Commands

Requires Node >= 18 (relies on the built-in global `fetch`).

Tests: `npm test` (runs `node --test` over `test/*.test.js` — zero dependencies, uses the built-in `node:test` runner). The suite covers the pure functions with real logic and a history of bugs; add cases there when you touch `parseReceivers`/`toTimeSpan`/`parseWeekdays`/`buildSoundPayload`/`isNotAvailable`/`groupChangeSummary`/`parseArgs`. Anything you want to unit-test must be exported from its module (that's why `isNotAvailable`/`groupChangeSummary` in `render.js` and `parseArgs` in `cli.js` are exported despite being otherwise-internal).

There are two independent UX modes sharing the same `PreComClient`/render code, with two independent storage strategies — do not conflate them when adding features:

- **Interactive menu** (`src/menu.js`, launched via `precomcli` with no args, `precomcli menu`, or `precomcli.ps1`) — a loop that never writes to the user's home directory. Its session (bearer token etc.) lives only in `src/tempSession.js`'s per-process temp dir and is deleted on exit/Ctrl+C/Ctrl+Break/window-close.
- **One-shot subcommands** (`src/cli.js` — see its `HELP` constant for the authoritative, current list rather than trusting a copy here, it's grown past 25 commands) — for scripting. These persist the token (and a couple of small non-secret values like `sendBy`) across separate invocations via `src/config.js` (`~/.precomcli/config.json`).

If you add a new API-backed action, implement it once as a `PreComClient` method in `src/api.js` and a renderer in `src/render.js`, then wire it into *both* `src/menu.js` (as a `MENU_ITEMS` entry) and `src/cli.js` (as a `cmdXxx` + `case`) if it makes sense in both places — don't duplicate formatting logic between them.

## Keeping documentation in sync

Every command/feature change (new command, changed flags, changed menu flow, a bug fix that changes behavior) touches more than code — update all of these in the *same* change, not as a follow-up:

- `src/cli.js`'s `HELP` text — must list every one-shot command and flag that actually exists.
- `README.md` — the interactive menu's numbered listing must match `MENU_ITEMS` in `src/menu.js` exactly (including the count), and the "One-shot commands" section must match `HELP`.
- This file's `src/*.js` architecture bullets below — update the renderer/method list and the "wired up so far" endpoint list in the `docs/swagger-v2.json` bullet whenever an endpoint is added or removed.
- The "API notes" section below — the moment you discover a non-obvious API quirk (typo'd fields, unversioned/versioned mismatches, surprising response shapes, semantics that differ from what a field name implies), write it down there immediately. A future session has no other way to rediscover it short of hitting the same bug again.

A change isn't finished until `README.md`, this file, and the actual code (`HELP`, `MENU_ITEMS`, `PreComClient` methods) all agree with each other. Treat that drift as a bug, not a nice-to-have — it already happened once: the `receivers`/`templates`/`message` commands shipped without a `README.md` update.

## Architecture

- `src/cli.js` — argv parsing (hand-rolled `parseArgs`, no yargs/commander; supports `--flag value` and `--flag=value`, with a `BOOLEAN_FLAGS` set so switch-style flags never swallow the following token) and one-shot command dispatch, plus routing `menu`/no-args into `src/menu.js`. See its `HELP` constant for the current command list. Every authenticated command goes through the `authed()` helper (loads config, enforces the session check, returns `{ cfg, client }`) — use it for new commands so the auth check can't be forgotten.
- `src/menu.js` — the interactive menu loop (`runMenu`). Top level is 9 items (`Log in`/`Messages`/`Availability`/`Groups`/`Capcodes`/`Group changes`/`Node info`/`Reset password`/`Log out`); `Messages`/`Availability`/`Groups`/`Capcodes`/`Group changes` open a submenu via the shared `runSubmenu(title, items)` + `promptAndDispatch(items, backLabel)` pair — reuse those for any new submenu rather than hand-rolling another loop. Owns its own action handlers (`actionXxx`) and reads/writes session state exclusively through `tempSession`, never `config` - **except** `actionResetPassword`, which (like `resetPassword` in `api.js`) deliberately doesn't require a logged-in session, since password recovery has to work when you can't log in.
- `src/api.js` — `PreComClient`: thin wrapper around `fetch` for the PreCom REST API. Add new endpoint methods here, not in `cli.js`/`menu.js`. `request()` takes `{ query, body, auth }` — pass `auth: false` for the rare unauthenticated endpoint (only `resetPassword` so far) rather than hand-rolling a separate `fetch`, so error-detail surfacing stays uniform. Also exports parsing/formatting helpers used by both UX modes: `parseReceivers` (`type:id[:label]` pairs), `toTimeSpan` (whole-hour integer → `"hh:mm:ss"`), `parseWeekdays` (`"mon,wed,fri"` → the bit-array integer `UpdateUserSchedulerPeriod` expects), `buildSoundPayload` (merges changes into the full 10-field `UpdateUserSound` object — used by both the `sound` command and the menu so the merge logic isn't duplicated), `SOUND_CATEGORIES` (the 5 categories → their `Sound*`/`CriticalAlerts*` field names), and `VALID_SOUNDS` (the 16 valid sound values, only documented in a swagger parameter description, nowhere else).
- `src/render.js` — output formatting shared by both modes; each `renderXxx` takes an `{ json }` option, and each `renderXxxPicker` (no `json` option) prints a plain numbered list for the interactive menu's "type a number" selection pattern (see `pickByNumber` in `menu.js`). Add new renderers here rather than inlining `console.log`/`JSON.stringify` calls in `cli.js` or `menu.js`.
- `src/store.js` — generic JSON-file-backed key/value store (`createStore(filePath)` → `{load,save,clear}`). Backs both `config.js` and `tempSession.js`.
- `src/config.js` — the **persistent** store at `~/.precomcli/config.json`, used only by the one-shot subcommands.
- `src/tempSession.js` — the **ephemeral** per-process store under `%TEMP%\precomcli-sessions\<pid>-<timestamp>\session.json`, used only by the menu. Also owns `registerCleanupHandlers()` (hooks `exit`/`SIGINT`/`SIGBREAK`/`SIGHUP`/`SIGTERM`/`uncaughtException` to synchronously delete its dir) and `sweepStaleSessions()` (called at menu startup; deletes leftover session dirs whose owning PID is no longer alive — the fallback for when a hard kill skipped the normal cleanup handlers).
- `src/prompt.js` — a **single shared** `readline.Interface` (`ask`/`closePrompt`) reused across all prompts in a run, plus a raw-mode masked-password reader (`askHidden`) that pauses/resumes that shared interface while it's active. Do not revert to creating a new `readline.Interface` per prompt — that silently drops any stdin input already buffered ahead of the current line. Note for anyone testing this interactively via piped/non-TTY stdin (e.g. an agent driving it non-interactively): a *burst* of piped lines reliably delivers only the first one to sequential `ask()` calls - Node's readline needs a tick between writes to re-arm, so a real human typing works fine but a single `printf 'a\nb\n' | ...` does not. Space piped writes out (e.g. a shell loop with a short `sleep` between each) if you need to drive multiple prompts non-interactively.
- `src/format.js` — low-level plain-text table/key-value printers (no chalk/table dependency), consumed by `render.js`. `printTable` collapses whitespace runs (including embedded newlines) in every cell — some upstream free-text fields (e.g. capcode descriptions) have been observed containing literal newlines that would otherwise break row alignment.
- `test/*.test.js` — `node:test` unit tests for the pure functions (run via `npm test`). No network, no fixtures — just the parsing/formatting logic. Requiring `src/cli.js` for `parseArgs` tests is side-effect-free (no readline spun up, no fs writes at import).
- `web/` — a third UX mode: a zero-dependency PWA (plain HTML/CSS/JS, no framework, no build step) published to GitHub Pages by `.github/workflows/pages.yml` (repo setting "Pages → Source: GitHub Actions" must be enabled once). Browser counterpart of `src/api.js`/`src/render.js` logic — it can't `require()` them, so the small bits it needs (`isNotAvailable`'s typo'd-field rule, the error-detail surfacing in `api()`, `CONTROL_ID_LABELS`) are duplicated in `web/app.js` with comments pointing back; if you change one of those rules in `src/`, change it there too. Covers a subset of features (status/available/not-available, groups with per-role member status and a member detail modal (info + direct message), alarms + respond, messages inbox/send, capcodes); session token lives in localStorage only. `DEFAULT_PROXY` in `app.js` is the shared relay every user goes through unless they set an override (login screen's Advanced section / Settings) — keep it in sync with the actually-deployed Worker URL (currently `https://precomcli.frosty-lake-b494.workers.dev`, deployed by Cloudflare Workers Builds from `worker/` on push). Two hard rules in `app.js`: never render API data via `innerHTML` (everything goes through the `el()` helper — API free-text fields are untrusted), and keep the strict CSP meta tag in `index.html` intact. Bump `CACHE` in `web/sw.js` when shipping changed shell files, or installed PWAs keep serving stale ones.
- `worker/worker.js` — the Cloudflare Worker CORS proxy the web app requires (PreCom sends no CORS headers, verified live 2026-07-22 — browsers cannot call it directly). Stateless by design: hardcoded upstream, exact endpoint+method allowlist (`ENDPOINTS` — must be extended when the web app starts using an endpoint not yet listed), origin allowlist, Bearer-required for `/api`, only `Authorization`/`Content-Type` forwarded, no storage, no logging. Do not weaken any of these to "simplify" — each blocks a specific abuse (open-proxy relaying, quota burn, credential exposure). Deployed manually (dashboard paste or `npx wrangler deploy`), not by CI.
- `precomcli.ps1` — thin launcher (`node bin\precomcli.js @args`) so the tool can be run directly in a PowerShell window (double-click / `.\precomcli.ps1`) without typing the `node bin/precomcli.js` path.
- `CHANGELOG.md` — keep it current when adding/changing commands (same discipline as README/HELP). `.editorconfig` — the canonical style (2-space, LF, final newline); match it.
- `docs/swagger-v2.json` — cached copy of the upstream Swagger 2.0 spec (`GET /Mobile/swagger/docs/v2`). Consult this before adding new commands rather than re-fetching it. There are 48 distinct capabilities once you dedupe the versioned/unversioned and multi-controller (e.g. `/api/User/GetUserCapcodes` vs `/api/Capcode/GetUserCapcodes` are the same operation) duplication described in the "API notes" section below - **use `operationId`'s suffix after the underscore to check for duplicates, not the path, if you're ever unsure whether something is already wired up.** 43 are wired up: everything through Phase 2, plus all 11 group-change endpoints (`group-change*` commands / "Group changes" submenu) and 2 read-only additions, `getPiketSchedule`/`getShiftAppointments` (`piket-schedule`/`shifts` commands, "On-call schedule" under Groups, "View shift work" under Availability). Not yet wired: Piket's 4 write endpoints (`AddScheduleItem`/`UpdateScheduleItem`/`DeleteScheduleItem`/`TakeScheduleItem`) and shift work's 2 write endpoints (`UpdateShiftAppointments`/`ImplementShiftAppointments`) - both blocked on the same problem: swagger gives Piket zero description text, and `ShiftWorkDay`'s raw integer fields (`ShiftType`, `Day`, `MomentNbr`, `TimeFrom`, `TimeTo`) are undocumented, and the account used to investigate so far has neither on-call slots nor shift-work data configured to decode either against. Confirmed live (2026-07-22): both read endpoints work (`GetSchedule` 200s with `[]`, or 401 for a group the account lacks Piket rights to; `GetShiftAppointments` 200s with an all-zero "nothing configured" object) - the blocker is genuinely a lack of real data, not a broken endpoint. Next step if picking this back up: either wait for the account to have real slots/shifts, or get explicit go-ahead for a live create-then-delete test (see the git history around 2026-07-22 for the exact tradeoff discussion). Deliberately excluded: `SetGeofencePosition` (needs real GPS, meaningless from a desktop CLI), the `Registration` family (push-notification device registration - this CLI doesn't receive pushes), `GetUserMenu` (mobile app UI navigation structure, not useful CLI data).

## API notes (not obvious from the spec itself)

- Auth is OAuth2 Resource Owner Password Credentials: `POST {baseUrl}/Token` (form-encoded `grant_type=password&username=&password=`) — **not** documented as a path in the swagger spec itself (only referenced obliquely via `securityDefinitions.oauth2.authorizationUrl`, which points at a stale `localhost` dev URL). Authenticated calls then use `Authorization: Bearer <access_token>`.
- Every endpoint exists in both an unversioned form (`/api/User/...`) and a versioned form (`/api/v{version}/User/...`, currently `v2`). This CLI intentionally uses the unversioned routes for simplicity.
- Error bodies are JSON like `{"Message": "..."}` on failure but can also come back with an empty body (e.g. 403 with `Content-Length: 0`) — `PreComClient.request` and the error handling in `cli.js` account for both.
- `GetOccupancyLevels` returns an object keyed by ISO date string → integer level (positive = fully staffed, negative = short-staffed, 0 = exact), not an array.
- Messaging is the one area where the unversioned/versioned pairing in the previous bullet breaks down: the unversioned `/api/Msg/{GetReceivers,GetTemplates,SendMessage}` live under a separate `PreComMsg`-tagged controller, distinct from the versioned `/api/v{version}/PreComMessage/...` controller — same request/response shapes, but they are not just "drop the version prefix" of each other. This CLI uses the unversioned `Msg` controller, consistent with the rest of the codebase.
- **The availability model, fully decoded via live A/B + write tests 2026-07-22** (comparing the official app's state against flags, then round-tripping writes on a live account). The model: (1) `GetUserSchedulerAppointments` returns your **effective availability (on-call) timeline** — blocks are when you ARE available. The endpoint's own swagger summary ("all unavailable appointments") is WRONG about the response; live-confirmed. (2) The write endpoints are **RANGE operations on that timeline, and their direction matches their names, not the read endpoint**: `AddUserSchedulerAppointment(date, from, to)` marks the range NOT available (punches a hole in the blocks — confirmed: the official app's "unavailable until 17:00" split a 10:15–24:00 block into 10:15–15:00 + 17:00–24:00), and `DeleteUserSchedulerAppointment(date, from, to)` clears not-available markings in the range so availability returns (confirmed: deleting 15:00–17:00 re-merged the block to 11:00–24:00). Delete is NOT an exact-match record delete — any range works. (3) `NotAvailalbeScheduled` (the `Availalbe` typo is PreCom's, in the live spec) is **inverted from its name: TRUE = a scheduled availability block is active now = the user IS available**. Live: available in app → `{NotAvailable: false, NotAvailalbeScheduled: true}`; unavailable → both false. `isNotAvailable` in `render.js` (duplicated in `web/app.js` — keep in sync) implements: manual `NotAvailable` → unavailable; else unavailable iff the scheduled flag is false; flag absent → manual toggle only. (4) `SetAvailable` does NOT clear schedule-driven unavailability (confirmed live — 200, no effect); `PreComClient.makeAvailable()` is the working composite (manual toggle via `SetAvailable` + `Delete(currentHour → 24:00:00)` for the schedule part) — use it, not bare `setAvailable()`, for "make me available" features. (5) Practical write rules: whole hours only (`hh:mm:ss` TimeSpans; `24:00:00` is a valid end-of-day `to` — see `toEndTimeSpan`); the v2 `/api/v2/SchedulerAppointment/*` twin controller exists on this host and returns identical data, but its `HH:MM` param format from the companion HA project **silently no-ops here** (200, no change) — use the unversioned routes; **any write normalizes that whole day's blocks to whole hours server-side** (observed destructively: an `00:00–08:15` block became `00:00–08:00`, a 15-minute block vanished) — mention this in user-facing flows. (6) `https://app.pre-com.nl` (the companion `precom-homeassistant` project's host) is a separate auth realm — Mobile-API tokens get 401 there; that project's `api.py` has the correct Add/Delete direction but reads the `NotAvailalbeScheduled` flag with the inverted (wrong) interpretation in its verification step. This finding explains and supersedes the earlier "status disagrees in both directions / propagation delay" caveat — it was flag inversion all along.
- **`SendMessage`'s `SendBy` field is the sender's PreCom-internal sender ID, NOT their `UserID` — and it is not derivable from any Mobile API endpoint.** Root-caused 2026-07-22 via live testing (see below): sending `SendBy: <own UserID>`, or omitting it, both 500 with a generic ASP.NET `{"Message":"An error has occurred."}` (unhandled exception, detail suppressed - `ExceptionMessage`/`ModelState` were both absent from the body every time). This is the *entire* cause - the documented `Receiver{Type,ID,Label}` shape, `Priority`/`Response`/`ValidFrom`/`CalculateGroupID`, and both the unversioned/versioned controllers were all confirmed fine and never needed changing. `SendBy` was found by capturing a real, working request from the separate browser portal (`https://portal.pre-com.nl`, cookie/session auth, entirely different from the Mobile API's OAuth bearer token) sending a message to a known account: it used `sentBy=64` where the account's own `UserID` is `22548` - a completely different number, and not equal to `UserID`, `HomeNodeID`, or any `GroupID` the account belongs to or can see. It's likely an identifier from a separate "operator/login" identity table the portal uses that the Mobile API has no endpoint for. **Consequence: this CLI cannot auto-populate `SendBy` for any account.** It must come from the user once (`--send-by <id>` / `PRECOM_SEND_BY` env var / interactive prompt in `cli.js`, cached in `~/.precomcli/config.json`; a per-session prompt cached in `tempSession` for `menu.js`) - see the README for how a user finds their own value (capture one portal request the same way this was diagnosed). If a future PreCom API update exposes this value through the Mobile API, replace the manual prompt with that instead.
- The portal's `PreComMsgLog/Submit` endpoint (`https://portal.pre-com.nl/PreCom/PreComMsgLog/Submit`, form-encoded, cookie-session auth) is a parallel, undocumented implementation of message sending, entirely separate from the Mobile API's `SendMessage` and not in the swagger spec at all. Its receiver shape also differs (`{Text, Value, Type, UserID, GroupID, PreComMsgGroupID, Disabled}` vs. the Mobile API's `{Type, ID, Label}`) but per the live tests above, the Mobile API's simpler shape works fine as long as `SendBy` is correct - so this CLI has no reason to target the portal's endpoint instead.
- `GetAlarmMessages(msgInID, previousOrNext)` is a paging cursor, not a filter: `previousOrNext: 0` means "ignore `msgInID`, give me the most recent alarm" (confirmed live) - that's the useful default for "what's my latest alarm", which is what `alarms`/"Alarm history" use. Negative/positive page backward/forward *from* `msgInID`.
- `GetUserSchedulerAppointments`'s `Duration` field is not always a clean `"hh:mm:ss"` whole-hour string, even though `AddUserSchedulerAppointment`'s swagger description says "only whole hours" for what you can *add*. Confirmed live: existing blocks can have non-zero minutes (`"08:15:00"`) and multi-day spans use a `"d.hh:mm:ss"` format (`"1.00:00:00"` = 1 full day) - almost certainly from blocks created via the app/portal, which aren't restricted to whole-hour single-day spans the way this CLI's `add`/`schedule-add` are. Since `DeleteUserSchedulerAppointment` turned out to be a range-clear rather than an exact-match delete (see the availability-model bullet above), odd-duration blocks pose no reconstruction problem — both `schedule-remove` and the menu's "Clear not-available markings" just take an explicit whole-hour range from the user.
- Tier 1 write endpoints added in that pass (`SetAvailable`, `SetAvailabilityForAlarmMessage`, `AddUserSchedulerAppointment`, `DeleteUserSchedulerAppointment`, `UpdateUserCapcode`) - and the Phase 2 ones added after it (`UpdateUserSchedulerPeriod`, `SetOutsideRegion`, `UpdateUserSound`) - are implemented to the documented schema but were deliberately **not** live-tested the way `SendMessage` was, since each has real operational consequences on a live dispatch account (disabling a real capcode, deleting the wrong schedule block, marking yourself available for real alarms, changing real recurring availability) - unlike a `SendMessage` test, these aren't "send yourself a harmless text." If one of these turns out to 500 the way `SendMessage` did, expect a similar hidden-field problem and use the same live-diagnostic approach (self-only test calls, compare against a captured working request) to root-cause it - don't just guess at fixes.
- `SetOutsideRegion`'s body is much simpler than its name suggests: despite "Set current geofence position" existing as a *separate* endpoint (`SetGeofencePosition`, real lat/long, excluded from this CLI), `SetOutsideRegion`'s `Region` model is just `{ Location: { Geofence: "ENTER" | "EXIT" } }` - a string, not coordinates. Confirmed by reading the model (`WebApi_PreCom.Models.Region`/`Location`), not live-tested.
- `Group/GetAllFunctions(groupID, date)` returns the full `Group` object (not just functions) with `ServiceFuntions: [{ServiceFunctionID, Label, NumberNeeded, Users: [...]}]` nested inside - confirmed live, real data renders cleanly (fire-brigade role names like "Bevelvoerder"/"Chauffeur"/"Manschap" with assigned user lists). This is also the endpoint that will be needed to resolve `serviceFunctionId` values for the still-unbuilt Piket endpoints. **The embedded `Users` are partial User objects**: `NotAvailalbeScheduled` is NOT populated there (false for every member — confirmed live 2026-07-22), so `isNotAvailable()` would misreport the whole group as unavailable. Instead each member carries `SchedulerDays`, keyed by the requested date (`"2026-07-22T00:00:00"`), with `Hour0`..`Hour23` booleans where **true = available that hour** (verified against a known-available account's own schedule) — combine that with the still-meaningful manual `NotAvailable`, as `isMemberAvailable` in `web/app.js` does.
- `Account/ResetPassword` and the menu's "Reset password" deliberately bypass this client's normal auth requirement (`PreComClient.resetPassword` does a raw `fetch`, not `request()`) - password recovery has to work when the user can't log in at all. Don't "fix" this to require a token.
- Group changes follow a consistent `Add*(groupId, ...)` vs. `Update*(groupUserID, groupId, ...)` pattern across all 3 variants (Days/Period/Periodically) - confirmed by reading all 6 swagger definitions, not just the Add ones. `Add*` operates on the account's implicit "first" group change; `Update*` targets a specific one by its `groupUserID` (from `GetGroupChange`/`GetAllGroupChanges`). The CLI's `group-change-days`/`-period`/`-recurring` commands and the menu's "Add a group change" consolidate this into one command/flow each: no `--group-user-id` → `Add*`, with one → `Update*`. This halves the command count (3 instead of 6) without losing any capability - keep this pattern if extending group-change support further.
- `GroupChangePeriodically`'s `startTime`/`stopTime` are full date-time strings, NOT the whole-hour `TimeSpan` strings (`toTimeSpan`) that `UpdateUserSchedulerPeriod`'s `from`/`to` use, despite both being "recurring time window" concepts on neighboring endpoints. Confirmed by reading both swagger definitions side by side - don't assume the scheduler-appointment helpers apply here.
